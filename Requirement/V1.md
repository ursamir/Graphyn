# Final Product Requirements

## 1. Product Vision

The platform shall evolve from an audio-focused dataset pipeline tool into a:

```text id="xv2h9p"
General-purpose AI/workflow execution platform
```

capable of powering:

* AI pipelines
* ML training/inference workflows
* edge AI workflows
* multimodal processing graphs
* automation workflows
* agent-generated workflows
* plugin ecosystems

The system shall remain:

* SDK-first
* CLI-first
* API-first
* MCP-native
* frontend-optional

The existing architecture already strongly supports this direction through:

* typed node system
* DAG executor
* plugin auto-discovery
* SDK/API/CLI layers
* graph-based execution runtime
* structured metadata system
* workspace/runtime abstraction 

---

# 2. Existing Architecture Preservation

The following current architectural strengths must remain foundational:

* Enhanced Node System
* AutoDiscovery registry system
* Typed ports and compatibility checking
* DAG execution engine
* YAML graph serialization
* RunManager and workspace system
* Plugin architecture
* Structured event streaming
* SDK + CLI support
* REST API layer
* Category-based node organization

These are already implemented and must not be regressed.

---

# 3. Core Architecture Requirements

## 3.1 SDK as Source of Truth

The Python SDK shall become the canonical platform interface.

All interfaces must delegate to the SDK:

* CLI → SDK
* REST API → SDK
* MCP → SDK
* Frontend → API/SDK

Business logic duplication across interfaces is prohibited.

The existing `Pipeline` and `PipelineNode` abstractions shall evolve into the primary public runtime API. 

---

## 3.2 Frontend Independence

The execution engine must remain fully usable without any frontend.

All workflows must be executable through:

* Python SDK
* CLI
* REST API
* MCP

The React frontend shall remain an optional orchestration layer only.

---

## 3.3 Graph Runtime

The platform shall continue using DAG-based graph execution.

The runtime shall support:

* DAG execution
* branching
* merging
* conditional execution
* streaming execution
* partial execution
* resumable execution
* async execution
* parallel execution

The current `PipelineGraph` and `NodeExecutor` systems shall be extended rather than replaced. 

---

# 4. Intermediate Representation (IR)

The platform shall define a formal graph IR independent of execution runtime.

The IR shall:

* be serializable
* support versioning
* support validation
* remain runtime-agnostic
* support deterministic replay

The current YAML representation shall evolve into:

* canonical graph IR
* SDK graph objects
* API transport schema
* MCP graph schema

The IR must support:

* local execution
* distributed execution
* edge execution
* container execution
* remote workers
* future WASM runtimes

---

# 5. Node System Requirements

The existing Enhanced Node System shall become the primary platform abstraction. 

---

## 5.1 Declarative Nodes

Nodes shall become fully declarative and self-describing.

Each node must expose:

* node_type
* version
* category
* tags
* input schema
* output schema
* parameter schema
* capability metadata
* runtime requirements
* hardware compatibility

Current `NodeMetadata`, `InputPort`, `OutputPort`, and `NodeConfig` structures shall be extended to support this. 

---

## 5.2 Capability Metadata

Nodes should support:

* requires_gpu
* supports_cpu
* supports_edge
* deterministic
* cacheable
* streaming_support
* realtime_support
* memory_requirements
* dependency_requirements
* batch_support

This metadata shall be machine-readable for:

* MCP
* agents
* schedulers
* optimizers
* deployment planners

---

## 5.3 Typed Data Contracts

The platform shall continue enforcing strongly typed data flow.

Existing `PortDataType` architecture shall remain core.

The platform shall standardize types such as:

* AudioSample
* DataSample
* FeatureArray
* TensorBatch
* ModelArtifact
* TFLiteArtifact
* PredictionResult
* DeploymentArtifact

Untyped dict-based execution should be minimized.

---

## 5.4 Category-Based Node Organization

Current category-folder architecture shall remain standard:

* `audio/`
* `ml/`
* future domains

Framework/runtime logic must remain separated from node implementations.

Future categories may include:

* vision/
* llm/
* robotics/
* edge/
* automation/
* networking/
* sensors/
* deployment/

---

# 6. Plugin System Requirements

The existing AutoDiscovery plugin architecture shall evolve into a full plugin ecosystem.

Plugins shall support:

* automatic discovery
* isolated dependencies
* semantic versioning
* capability discovery
* hot registration
* metadata indexing

Future-compatible architecture should support:

* pip-installable plugins
* plugin manifests
* sandboxed plugins
* remote plugins
* signed plugins

---

# 7. Runtime Requirements

## 7.1 Execution Runtime

The runtime shall support:

* synchronous execution
* asynchronous execution
* event-driven execution
* streaming execution
* retries
* cancellation
* checkpointing
* resumability

Existing:

* `NodeExecutor`
* `RetryPolicy`
* `PipelineLogger`
* `RunManager`
* checkpoint/cache systems

shall remain foundational.

---

## 7.2 Runtime Backends

The architecture shall remain runtime-backend independent.

Future execution backends may include:

* local Python runtime
* Docker runtime
* distributed workers
* Kubernetes runtime
* edge runtime
* GPU runtime
* WASM runtime

Node logic must avoid assumptions tied to a single runtime.

---

## 7.3 Event Streaming

Structured runtime events shall remain first-class.

Current NDJSON event streaming shall evolve into:

* SDK event streams
* WebSocket streams
* MCP event streams
* distributed execution monitoring

Existing structured events must remain compatible:

* pipeline_start
* node_start
* node_end
* node_error
* pipeline_summary
* done/error

---

# 8. SDK Requirements

The SDK shall expose:

* graph creation
* graph validation
* graph execution
* node discovery
* runtime inspection
* plugin management
* artifact management
* event subscriptions

The SDK shall remain:

* type-safe
* scriptable
* composable
* agent-friendly

Current SDK abstractions shall be preserved and expanded. 

---

# 9. CLI Requirements

The CLI shall become a production-grade orchestration interface.

It shall support:

* workflow execution
* graph validation
* graph inspection
* runtime debugging
* run management
* artifact inspection
* plugin management
* registry inspection

The CLI must remain fully functional without the frontend. 

---

# 10. API Requirements

The REST API shall remain thin orchestration around SDK/runtime systems.

Existing `/api/v1/` architecture shall remain canonical.

The API shall expose:

* node discovery
* graph validation
* workflow execution
* streaming execution
* runtime monitoring
* project management
* artifact access
* plugin discovery

Future support may include:

* gRPC
* WebSocket execution streams
* distributed orchestration APIs

---

# 11. MCP Requirements

MCP support shall become first-class architecture.

The MCP layer shall expose:

* node schemas
* capability metadata
* graph schemas
* runtime interfaces
* execution APIs
* artifact APIs
* validation APIs

Agents must be able to:

* discover nodes
* generate workflows
* validate graphs
* execute workflows
* inspect outputs
* optimize execution

without frontend interaction.

The node system and graph IR shall be designed specifically for machine operability.

---

# 12. Workspace & Artifact System

The existing workspace architecture shall remain foundational.

The system shall support:

* datasets
* tensors
* models
* deployment bundles
* checkpoints
* metrics
* logs
* inference outputs

Artifacts should support:

* lineage tracking
* metadata
* reproducibility
* versioning
* caching

---

# 13. Provenance & Reproducibility

The system shall track:

* workflow lineage
* node execution history
* dependency versions
* runtime metadata
* graph snapshots
* artifact provenance

Existing `RunManager` and metadata tracking systems shall evolve into a full provenance layer.

---

# 14. Edge AI Requirements

The platform shall support Edge AI workflows.

Potential deployment targets:

* Raspberry Pi
* NVIDIA Jetson
* Coral TPU
* STM32 AI
* ESP32
* Android NNAPI
* TensorRT
* ONNX Runtime
* TFLite

The graph system should support:

* quantization
* model conversion
* hardware optimization
* deployment packaging
* inference benchmarking

Existing ML node architecture shall evolve toward deployment-oriented workflows. 

---

# 15. Known Technical Debt & Cleanup

The following issues should be addressed progressively:

* async status dual-source tracking
* thread-safe ingestion progress
* incomplete cache coverage
* legacy router cleanup
* plugin dependency management
* stronger runtime isolation
* sink node return consistency

Current known issues are documented and acknowledged. 

---

# 16. Long-Term Strategic Goal

The primary strategic asset of the platform shall become:

```text id="7d2j1f"
A standardized executable workflow graph ecosystem
```

not merely:

* a frontend application
* an audio processing tool
* a node editor

The long-term moat shall be:

* graph/runtime specification
* SDK ecosystem
* plugin ecosystem
* typed execution model
* MCP operability
* agent-native workflows

The platform should ultimately function as:

```text id="u2q8n5"
An AI-operable workflow operating system
```

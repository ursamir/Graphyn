FILE:        PluginPackage/Audio/environment_simulator/nodes.py
FUNCTION:    EnvironmentSimulatorNode._simulate
CATEGORY:    Edge Case
SEVERITY:    HIGH
--------------------------------------------------------------------
WHAT IT CLAIMS TO DO:
Simulate a physical acoustic environment by convolving audio with a RIR
generated by pyroomacoustics.

WHAT IT ACTUALLY DOES:
`e_absorption, max_order = pra.inverse_sabine(rt60, dims)` — `pra.inverse_sabine`
can raise `ValueError` if `rt60` is too small for the given room dimensions
(the Sabine formula produces absorption > 1.0, which is physically impossible).
For the "outdoor" preset, `rt60=0.05` and `dims=[50, 50, 10]`. The Sabine
formula gives `e_absorption = 0.161 * V / (S * rt60)` where V=25000 m³ and
S=6500 m². This gives `e_absorption ≈ 12.4`, which is > 1.0 and will cause
pyroomacoustics to raise `ValueError: Absorption coefficient must be between
0 and 1`.

EVIDENCE:
```python
e_absorption, max_order = pra.inverse_sabine(rt60, dims)
# For outdoor preset: rt60=0.05, dims=[50,50,10]
# Sabine: e_absorption = 0.161 * 25000 / (6500 * 0.05) ≈ 12.4 → ValueError
```

REPRODUCTION SCENARIO:
```python
node = EnvironmentSimulatorNode(config=EnvironmentSimulatorNode.Config(preset="outdoor"))
node.setup()
node.process([AudioSample(...)])  # ValueError from pra.inverse_sabine
```

IMPACT:
The "outdoor" preset (a documented, named preset) crashes on every call.
Users cannot use the outdoor preset without a workaround.

FIX DIRECTION:
For the outdoor preset, use a direct absorption coefficient instead of
`inverse_sabine`:
```python
if rt60 <= 0.1:  # very low RT60 — use near-anechoic absorption directly
    e_absorption = 0.99
    max_order = 1
else:
    e_absorption, max_order = pra.inverse_sabine(rt60, dims)
```
Or fix the outdoor preset dimensions to be physically consistent with rt60=0.05.

FILE:        PluginPackage/Audio/environment_simulator/nodes.py
FUNCTION:    EnvironmentSimulatorNode._simulate
CATEGORY:    Edge Case
SEVERITY:    HIGH
--------------------------------------------------------------------
WHAT IT CLAIMS TO DO:
Clamp mic and source positions to be inside the room.

WHAT IT ACTUALLY DOES:
`mic_pos = [min(mic_pos[i], dims[i] - 0.1) for i in range(3)]` — clamps
positions to be at most `dims[i] - 0.1` metres from the origin. However,
it does NOT clamp the minimum — positions can be 0.0 or negative. pyroomacoustics
requires all positions to be strictly inside the room (> 0 and < dims[i]).
A position of 0.0 or negative will cause `pra.ShoeBox.add_source()` or
`pra.ShoeBox.add_microphone()` to raise `ValueError: Source must be inside
the room`.

EVIDENCE:
```python
mic_pos = [min(mic_pos[i], dims[i] - 0.1) for i in range(3)]
src_pos = [min(src_pos[i], dims[i] - 0.1) for i in range(3)]
# No lower bound clamp — positions can be 0.0 or negative
room.add_source(src_pos, signal=y)  # ValueError if src_pos[i] <= 0
```

REPRODUCTION SCENARIO:
```python
node = EnvironmentSimulatorNode(config=EnvironmentSimulatorNode.Config(
    preset="custom",
    room_dimensions=[5.0, 4.0, 3.0],
    source_position=[-1.0, 2.0, 1.5],  # negative x
))
node.process([AudioSample(...)])  # ValueError from pyroomacoustics
```

IMPACT:
Crash for any custom preset with positions at or below 0.

FIX DIRECTION:
```python
mic_pos = [max(0.05, min(mic_pos[i], dims[i] - 0.05)) for i in range(3)]
src_pos = [max(0.05, min(src_pos[i], dims[i] - 0.05)) for i in range(3)]
```

FILE:        PluginPackage/Audio/environment_simulator/nodes.py
FUNCTION:    EnvironmentSimulatorNode._simulate
CATEGORY:    Edge Case
SEVERITY:    MEDIUM
--------------------------------------------------------------------
WHAT IT CLAIMS TO DO:
Convolve input audio with the generated RIR.

WHAT IT ACTUALLY DOES:
`room.add_source(src_pos, signal=y)` — if `y` is zero-length
(`sample.data = np.array([])`), pyroomacoustics may raise or produce an empty
simulation. `room.simulate()` on a zero-length signal may raise `ValueError`
or produce `room.mic_array.signals[0]` as an empty array. The subsequent
`y_sim[:max_len]` and normalisation would handle an empty array silently
(returning empty), but the pyroomacoustics call itself may raise.

EVIDENCE:
```python
y = sample.data.astype(np.float32)
# ...
room.add_source(src_pos, signal=y)  # may raise for zero-length y
room.simulate()
```

REPRODUCTION SCENARIO:
Pass `AudioSample(data=np.array([]))` to process().

IMPACT:
Crash or silent empty output for zero-length audio.

FIX DIRECTION:
```python
if len(y) == 0:
    log.warning("EnvironmentSimulatorNode: skipping zero-length sample %s", sample.path)
    return copy.deepcopy(sample)
```

FILE:        PluginPackage/Audio/environment_simulator/nodes.py
FUNCTION:    EnvironmentSimulatorNode._simulate
CATEGORY:    Silent Failure
SEVERITY:    MEDIUM
--------------------------------------------------------------------
WHAT IT CLAIMS TO DO:
Normalise output to prevent clipping.

WHAT IT ACTUALLY DOES:
```python
peak = np.max(np.abs(y_sim))
if peak > 1e-6:
    y_sim = y_sim / peak * np.max(np.abs(y))
```
This normalises the simulated audio to have the same peak amplitude as the
input. However, if the input `y` is silence (all zeros), `np.max(np.abs(y)) == 0`
and the normalisation produces `y_sim = y_sim / peak * 0 = all zeros`. The
output is silence regardless of the RIR. This is technically correct (silence
in → silence out) but may surprise users who expect the RIR to add reverb
to silence.

More importantly: if `y_sim` is all zeros (e.g. RIR convolution produced
silence), `peak <= 1e-6` and the normalisation is skipped. The output is
returned as-is (all zeros). No warning is emitted.

EVIDENCE:
```python
peak = np.max(np.abs(y_sim))
if peak > 1e-6:
    y_sim = y_sim / peak * np.max(np.abs(y))
# else: y_sim returned as-is (may be all zeros)
```

IMPACT:
Silent all-zero output if RIR convolution produces silence. No warning.

FIX DIRECTION:
```python
if peak <= 1e-6:
    log.warning("EnvironmentSimulatorNode: simulated signal is near-silent "
                "(peak=%.2e) — RIR convolution may have failed", peak)
```

FILE:        PluginPackage/Audio/environment_simulator/nodes.py
FUNCTION:    EnvironmentSimulatorNode.Config
CATEGORY:    Type Safety
SEVERITY:    LOW
--------------------------------------------------------------------
WHAT IT CLAIMS TO DO:
Store room configuration including room_dimensions, mic_position, source_position
as lists.

WHAT IT ACTUALLY DOES:
`room_dimensions: list = [5.0, 4.0, 3.0]` — typed as unparameterised `list`.
Pydantic will accept any list, including lists of wrong length (e.g. 2D instead
of 3D) or wrong types (e.g. strings). `pra.ShoeBox(dims, ...)` requires a
3-element list of positive floats. A 2-element list will raise `ValueError`
inside pyroomacoustics with a confusing traceback.

EVIDENCE:
```python
room_dimensions: list = [5.0, 4.0, 3.0]  # no length or type validation
mic_position: list = [2.5, 2.0, 1.5]
source_position: list = [1.0, 1.0, 1.5]
```

REPRODUCTION SCENARIO:
```python
Config(room_dimensions=[5.0, 4.0])  # 2D room — accepted by Pydantic
# pra.ShoeBox([5.0, 4.0], ...) → ValueError inside pyroomacoustics
```

IMPACT:
Confusing crash deep in pyroomacoustics for invalid config. No early validation.

FIX DIRECTION:
Add a Pydantic validator:
```python
@pydantic.field_validator("room_dimensions", "mic_position", "source_position")
@classmethod
def _three_positive_floats(cls, v: list) -> list:
    if len(v) != 3:
        raise ValueError(f"Must be a 3-element list [x, y, z], got {len(v)} elements")
    if any(float(x) <= 0 for x in v):
        raise ValueError(f"All dimensions must be positive, got {v}")
    return [float(x) for x in v]
```
